1. Executive Summary

What this is: AI-RUN SOS is a vertically integrated, AI-native staffing operating system purpose-built for the C2C/W2/Contract IT staffing market. It is not a CRM with AI bolted on. It is a system-of-record + workflow engine + agent runtime designed to replace 90–95% of human labor in the staffing lifecycle with deterministic AI agents, while preserving human judgment at high-stakes decision gates.

The core thesis: In IT contract staffing, the difference between a $2M/year shop and a $20M/year shop is not headcount — it is speed-to-submit, conversion discipline, and vendor intelligence compounding over time. This system codifies that thesis into software.

Target: 1 closure/day at $10+/hr margin = $5M/year gross margin potential
Endgame: 1 human operator + AI agents = output of 10 recruiters

The system creates a closed-loop operating system:

Email Inbox → Extraction → Trust Scoring → Matching → Submission → Follow-up → Response Detection → Trust Update

2. Business Context & Value Proposition

The Problem Being Solved

The US IT contract staffing market is a $180B+ annual market dominated by fragmented, labor-intensive operations. A typical bench sales operation:

  1. 11 recruiters monitor email inboxes manually
  2. Each receives 200–500 vendor emails/day with job requirements
  3. They manually scan, mentally filter, and decide which to act on
  4. Resume formatting takes 30–60 minutes per submission
  5. Follow-ups are forgotten or inconsistent
  6. Vendor trust is carried in individual recruiters' heads
  7. When a recruiter leaves, institutional knowledge walks out the door

Result: Industry-average conversion rates of 2–5% submission-to-placement.

Architecture Decisions (Why These Choices)

DecisionRationale
Prisma + raw SQL coexistPrisma handles CRUD. Analytics over 61M rows require hand-tuned SQL with CTEs and DISTINCT ON. Both coexist intentionally.
Regex extraction, not LLMsAt 812K emails, LLM extraction costs $8K–$40K per pass. Regex runs in minutes at zero cost. ~70–80% accuracy is sufficient because false negatives (missed reqs) are cheap.
In-memory cache, not RedisSingle-instance deployment uses Map<string, CacheEntry>. Redis available for multi-instance scaling.
PgBoss, not BullMQPostgreSQL-backed job queue eliminates Redis as a hard dependency for scheduling.

Agent Scoring Algorithms

MarginGuard — Margin Computation

grossMarginHr = billRate - payRate
burdenCost    = payRate × burdenPct
payrollTax    = payRate × payrollTaxPct
vendorCut     = billRate × vendorCutPct
portalFee     = billRate × portalFeePct
netMarginHr   = grossMarginHr - burdenCost - payrollTax - vendorCut - portalFee - otherFees

If netMarginHr >= $10/hr  → APPROVE
If netMarginHr >= $5/hr   → SOFT BLOCK (warning, override available)
If netMarginHr < $5/hr    → HARD BLOCK (CFO escalation required)

Extraction Algorithms

Email Classifier (Priority Order)

  1. Own domain → INTERNAL
  2. System/no-reply → SYSTEM
  3. Free email + resume patterns → CONSULTANT
  4. VMS domains (vectorvms, fieldglass) → CLIENT
  5. Client subject patterns (interview, PO, timesheet) → CLIENT
  6. Resume score ≥ 2 → CONSULTANT
  7. Req score ≥ 2 → VENDOR_REQ
  8. Company domain → VENDOR_OTHER

Req Signal Extractor

  • Subject patterns: c2c, w2, requirement, hiring, hot req
  • Body patterns: location:, duration:, rate:, client:, $/hr (needs ≥2 hits)
  • Extracts: title, location, rate text, employment type, skills
  • Links to: vendor_company + vendor_contact by domain

Actionability Score Computation

Composite score (0–100) from: employment type match (C2C/W2 get higher), rate presence, skill specificity, vendor trust, and location clarity.
Thresholds: 30 (display), 60 (premium) — operationally calibrated, not statistically derived.
